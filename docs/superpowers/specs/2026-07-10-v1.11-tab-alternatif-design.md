## Masalah

Tab "🎲 Kejutan (beta)" sekarang:
- **Random** — pilih acak dari pool Pareto yang belum ditampilkan 3 tab lain
- **Tidak deterministik** — hasil beda tiap refresh
- **Sering kosong** — kalau cuma ada 1 rute Pareto (contoh: Simpang Kuningan → Ragunan)
- **Lewatin rute nyata** — rute yang orang pakai tiap hari (jalan JPO → Underpass Kuningan 297m → route 6 one-seat ke Ragunan) **gak pernah muncul** karena router gak mau jalan >150m

Ground truth dari user (Reza, kerja di Simpang Kuningan): orang sering **jalan dulu** ke halte tetangga (via JPO, underpass) untuk dapetin:
- **Frekuensi lebih tinggi** (Underpass banyak trayek vs L13E jarang)
- **One-seat ride** (0 transfer vs 1 transfer dari halte persis)

Router sekarang **buta** ke perilaku ini karena `WALK_M = 150m` → jarak 297m ke Underpass gak dimodelin.

---

## Solusi: Tab "Alternatif" Deterministik

Ganti tab "🎲 Kejutan (random)" → **"Alternatif"** — rute seimbang yang:
1. **Deterministik** (hasil selalu sama untuk origin-dest yang sama)
2. **Beda** dari 3 pemenang (Tarif/Simpel/Jarak)
3. **Waras** (gak muter kebanyakan halte demi jalan sedikit)
4. **Boleh jalan lebih jauh** di titik awal/akhir buat akses halte yang lebih enak

### Studi Kasus (Simpang Kuningan → Ragunan)

**Sebelum** (router normal, seed cuma halte origin persis):
- Cuma ketemu **1 rute** (transfer di Denpasar/Patra)
- Tab Kejutan **kosong** (pool Pareto kosong)
- Rute Underpass (yang orang pakai) **gak muncul**

**Sesudah** (router mode-alternatif, seed halte dalam radius 400m):
```
tr=0 stops=9  jalan=297m | 6 @ Underpass Kuningan       ← JADI ALTERNATIF ✓
tr=1 stops=10 jalan=106m | 9 → 6B @ Denpasar
tr=1 stops=11 jalan=0m   | 13E → 6 @ Patra Kuningan
```
Rute Underpass (0 transfer) **muncul** dan **beda** dari 3 goal (yang semuanya 1 transfer).

## Desain Teknis

### 1. Isolasi Pencarian

Tab **Tarif/Simpel/Jarak tetap 100% byte-identik** — seed dari halte origin cost 0, **nol perubahan**.

Tab **Alternatif** = pencarian **terpisah** dengan:
- Seed origin: **semua halte dalam radius `ACCESS_M`** (default 400m) dengan cost awal = jarak jalan
- Egress dest: **ditunda** (v-nanti). Cukup seed origin dulu — kasus nyata (Simpang→Underpass) cuma butuh jalan di awal. Tambah egress kalau ada bukti kasus butuh jalan di ujung.
- Transfer tengah: tetep `WALK_M = 150m` (gak ngubah model transfer)

Isolasi → **nol regresi** di 3 tab lama, cuma Alternatif yang "berani jalan".

### 2. Parameter Tunable

```python
ACCESS_M = 400  # radius jalan akses/egress (meter), tunable via query/config
```

Tipikal jalan akses Jakarta:
- JPO Simpang→Underpass Kuningan: **297m** ✓ (masuk)
- Jalan >500m: mulai gak realistis buat komute harian

Konservatif: mulai 400m. Bisa dinaikkan kalau user request, atau dibuka sebagai slider UI nanti (v-masa-depan, YAGNI sekarang).

### 3. Guard "Waras" (Opsi A — Rasio Adaptif)

**Buang** rute yang haltenya **jauh lebih banyak** dari rute paling ringkas:

```python
fewest_stops = min(stops for _tr, stops, _wm, _path in pareto_candidates)
sane = [r for r in pareto_candidates if r[1] <= fewest_stops * 1.5]
```

Contoh kebunuh sampah:
- Rute normal: 9-11 halte
- Rute sampah: **63 halte** (5N Mikrotrans, muter gak jelas)
- 63 > 9×1.5 → **kebuang** ✓

Rasio 1.5 = toleransi 50%. Bisa di-tune nanti kalau ada bukti lapangan (sekarang cukup; prototype BRT-only gak muncul sampah setelah filter service class).

### 4. Filter Service Class

**BRT-only** default buat tab Alternatif (konsisten dengan v1.9 design: 8 kelas, BRT = prioritas tinggi).

```python
allowed = {"BRT"}  # route_desc = "BRT" → 35 rute koridor utama
```

Guard double:
1. Service class = BRT → buang Mikrotrans/non-koridor
2. Guard A → buang yang muter kebanyakan halte

Hasil prototype: **100% bersih**, gak perlu guard tambahan.

### 5. Seleksi Rute Alternatif

Dari kandidat yang lolos:
1. **Buang** yang path-nya **sama persis** dengan salah satu dari 3 pemenang goal
2. Sisa → **pilih 1** rute terbaik:
   - **Prioritas 1:** transfer paling sedikit
   - **Tie-break:** waktu tempuh (`etime`) tercepat
   - **Tie-break 2:** halte paling sedikit

Output: **1 rute** ditampilkan sebagai tab ke-4.

### 6. Behavior Kosong

Kalau gak ada rute waras yang beda dari 3 goal → **tab Alternatif disembunyikan** (gak dipaksa).

Jujur > maksa tampilkan rute duplikat/sampah.

### Backend (`route.py`)

**Fungsi baru:**
```python
def find_alternative(origin, dest, data, goals_result, radius=400, allowed={"BRT"}):
    """Cari rute alternatif dengan jalan akses/egress.
    
    Args:
        origin, dest: substring halte
        data: load() result
        goals_result: dict hasil find_goals() — buat deteksi duplikat
        radius: ACCESS_M (meter)
        allowed: set route_desc yang boleh (default BRT-only)
    
    Returns:
        (transfers, stops, path) atau None kalau gak ada kandidat waras
    """
    # 1. Seed origin-nearby stops dengan cost = jarak jalan
    # 2. Dijkstra goal="simple" (cost = halte×STOP_M + jalan)
    # 3. Kumpulin semua solusi Pareto (tr, stops, walk_m, path)
    # 4. Guard A: buang stops > min×1.5
    # 5. Buang yang path sama dengan goals_result values
    # 6. Pilih terbaik: min transfer, tie-break etime, tie-break stops
    # 7. Return (tr, st, path) atau None
```
