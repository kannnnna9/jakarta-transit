## 3. Sumbu & data

Semua data sudah tersedia setelah v1.6/v1.7 — **v1.8 tidak menambah field data.json baru.**

| Tab | Minimize | Sumber data | Sifat biaya | State pencarian |
|---|---|---|---|---|
| 💰 Tarif | rupiah total | `fare` (v1.7) | **stateful** (`brtPaid`) | perluas `(stop, route, brtPaid)` |
| ⏱️ Waktu | detik total | `etime` (v1.7) | aditif | biasa `(stop, route)` |
| 🚶 Jalan-kaki | Σ `xdist` (meter) | `xdist` (v1.6) | aditif | biasa `(stop, route)` |
| 🎲 Kejutan | — | kolam Pareto v1.5 | — | acak, UI-only |

Kenapa 3 sumbu ini beda-karakter (bukan duplikat satu sama lain):

- **Tarif ≠ Transfer.** BRT transfer dalam-sistem (`s`/`o`) gratis, jadi rute
  banyak-transfer bisa tetap Rp3.500. Sebaliknya rute 0-transfer via Royaltrans (PP,
  Rp20.000) itu mahal. Dan transfer `w` (jalan kaki keluar sistem) = tap ulang
  +Rp3.500 → kadang **lebih banyak transfer `s` malah lebih murah** dari 1 transfer
  `w`. Tarif menghindari layanan premium + menghindari keluar-gate.
- **Jalan-kaki ≠ Transfer.** Rute bisa banyak transfer `s`/`o` (pindah peron, nol
  jalan kaki) tapi Σ`xdist` = 0. Minim jalan-kaki khusus menekan meter `w`.
- **Waktu ≈ Jarak** (sebab itu "jarak" dibuang).

## 4. Router — 3 pencarian deterministik + kolam Kejutan

Aturan besi proyek tetap: **`route.py` (oracle) == `router.js`**, dibuktikan
`test-router.js`. v1.8 menambah **3 permukaan parity** (tarif, waktu, jalan-kaki).

### 4a. Fungsi pencarian terparametrisasi (DRY)

Satu fungsi shortest-path (Dijkstra label-setting) yang menerima fungsi biaya edge,
dipanggil 3x dengan biaya beda. Hindari 3 salinan algoritma. Terapkan pola identik di
`route.py` dan `router.js`.

- **Waktu:** biaya edge `ride (a→b)` = `etime[route][a][b]` detik. Transfer = 0 detik
  (ride-saja, sesuai keputusan v1.7 — TANPA waktu tunggu/jalan-kaki di angka). Aditif.
- **Jalan-kaki:** biaya edge transfer = `xdist` meter (khusus tipe `w`; `s`/`o` = 0).
  Ride = 0. Aditif.
- **Tarif:** biaya **stateful** — lihat 4b.

### 4b. Tarif — state diperluas `brtPaid`

Karena biaya BRT tergantung apakah sesi berbayar sedang aktif, node pencarian
diperluas jadi `(stop, route, brtPaid)`. Ini membuat tarif **aditif di graf yang
diperluas** sehingga Dijkstra tetap valid (optimum global terjamin).

Aturan biaya edge saat masuk leg baru (IDENTIK §5b spec v1.7 — jangan menyimpang):

```
# reset dulu di SETIAP leg (fix bug kurang-bayar):
if xtype == "w": brtPaid = false      # keluar sistem, sesi putus di leg mana pun

GR            → +0        (Mikrotrans; brtPaid tak berubah)
PP/PP2/PP3    → +price    (layanan terpisah; brtPaid tak diset)
FP/FP2 (BRT)  → brtPaid ? +0 : (+price, brtPaid=true)
```

Transfer `s`/`o` tak menyentuh `brtPaid` → sesi lanjut, transfer koridor gratis.
Verifikasi rumus ini SAMA PERSIS dengan `web/cost.js` v1.7 (satu sumber kebenaran
tarif; v1.8 memindahkannya ke dalam pencarian, bukan mengubah aturannya).

### 4c. Tie-break deterministik (TITIK RAWAN #1)

Banyak rute akan **seri** pada sumbu tujuan (mis. puluhan rute sama-sama Rp3.500).
Pemenang WAJIB dipilih dengan aturan pemecah-seri deterministik yang **identik di
Python & JS**, kalau tidak parity merah dan hasil goyang antar-perangkat.

Aturan pemecah-seri (urut, DECIDED):
1. biaya tujuan utama (rupiah / detik / meter) — terkecil menang.
2. jumlah transfer — lebih sedikit menang.
3. jumlah halte — lebih sedikit menang.
4. tie-break akhir stabil: bandingkan **urutan `stop_id` jalur** secara leksikografik
   (deterministik, tak bergantung urutan iterasi hash/dict).

Implementasi: JANGAN mengandalkan urutan iterasi dict/Map (beda Python vs JS). Selalu
`sort` eksplisit dengan kunci majemuk di atas sebelum ambil pemenang.

### 4d. Kolam Kejutan

Router tetap menghitung kumpulan Pareto v1.5 (non-dominated `(transfer, halte)`).
Kejutan = pilih acak satu anggota kolam yang **bukan** salah satu dari 3 pemenang
deterministik. Pemilihan acak dilakukan **di UI (`app.js`/`router.js` layer atas)**,
BUKAN di fungsi pencarian yang diuji parity. `route.py` tak ikut mengacak — dia oracle
untuk yang deterministik saja. Kalau kolam habis (semua anggota sudah jadi pemenang),
tab Kejutan disembunyikan.

## 5. Oracle parity — `route.py` + `test-router.js`

- `route.py` implement 3 pencarian identik (§4) + hitung skor tampil (`~N mnt · RpX ·
  🚶Nm`) per pemenang.
- `test-router.js` tambah kasus per sumbu:
  - **Tarif:** rute dengan Royaltrans(PP) langsung vs BRT muter → pemenang tarif =
    BRT Rp3.500 (bukan PP Rp20.000). Rute 2-koridor BRT via `s` = Rp3.500. Via `w` =
    Rp7.000.
  - **Waktu:** edge delta arrival dikenal → pemenang = jumlah `etime` terkecil.
  - **Jalan-kaki:** dua rute, satu punya `w` 200m, satu semua `s` (0m) → pemenang
    jalan-kaki = yang 0m walau haltenya lebih banyak.
  - **Tie-break:** dua rute biaya sama → pemenang deterministik sesuai §4c, dan
    **Python == JS** (kasus cermin).
- Kejutan TIDAK diuji parity (non-deterministik, by design).
